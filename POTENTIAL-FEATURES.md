# Blick: Potential Features

Ideas under consideration, not yet committed to the build. Each entry notes the rough approach and the main tradeoff so we can judge priority later. Shipped functionality lives in FEATURES.md.

Blick's goal is a full personal Microsoft 365 client across every Apple surface — see and act on presence, meetings, mail, and Teams; read, reply, and initiate — with nothing leaving the user's own devices. The privacy posture is the only hard boundary: no backend, no third-party SDK, no new external destination, no token movement off the device that obtained it.

## Voice catch-up: interactive read-aloud and voice reply (in-app surface)

Phase 1 shipped on Siri: "Catch me up in Blick" (via `WorkdaySummaryIntent`) now reads the unread queue aloud as one capped block after a spoken "Want me to read them?" (see FEATURES.md). What Phase 1 proved on-device is that Siri's App Intents dialog **cannot** carry the interactive part of this feature: the framework identifies each prompt by its source call site, so a dynamic per-item loop ("read one, then let me reply / mark-read / next") collapses to a single identity and repeats endlessly. That is structural, not a bug. So the block read is as far as the Siri surface goes.

The interactive experience therefore moves to an **in-app read-aloud surface**: a "Catch me up" view driven by `AVSpeechSynthesizer` that narrates the queue item by item with continuous speech and on-screen Next / Reply / Mark-read controls, pausing between items and letting you act on each. This is where per-item pacing, a graceful stop, and the eventual **voice reply** all actually work, because we own the loop instead of the dialog framework owning it. It stays in posture — the phone already holds the mail and chats, the voice layer only reads what is fetched and routes replies / mark-reads through the same `Inbox` paths the app uses, no new Graph scope, no new destination. The trade against Siri is that it is tap-to-start rather than hands-free from a locked phone; the division of labor is Siri as the fast "read me what I missed" entry, the app as where you act on it. Phase it: item-by-item read with Next/stop first, then Reply (dictation via the keyboard mic) and Mark-read. Relationship to the Earshot concept ([[teams-voice-app-concept]]): that is ambient, per-contact Teams voices as messages arrive, while this is on-demand "read me what I missed" triage, different enough to belong in Blick.

## Curated people list and message filtering

A short, user-starred set of the people who matter most, held on the device. Its first payoff is catch-up: filter the mail and Teams streams Blick already fetches down to just those people, so "what did my key people send today" is one tap. The filtering needs no new Graph scope, because it happens locally over messages already in hand, matched by sender identity. The only new state is the starred subset itself, stored on-device (UserDefaults, or the app group so the widget can read it later) as a light reference — a display name plus email or UPN — with no backend and nothing synced off the device, so the privacy posture is unchanged.

Getting a person onto the list is easy on iOS: pick them with `CNContactPickerViewController`, which runs out of process and returns only the contact the user tapped, so it needs no Contacts permission and no `People.Read` scope. Starring someone straight from a message they already sent is the other entry path, since the fetched streams carry sender identities. This starred set is a convenience for filtering, not a prerequisite for the compose and initiate features below, which select recipients through the same Contacts picker directly.

## Initiate a Teams chat (new message)

A compose surface that starts a new Teams chat rather than only replying into threads that already carry pending activity. Pick a person with the system Contacts picker, type a message, send, with a group chat when more than one person is chosen. Posting the first message is `POST /chats/{id}/messages`, covered by the `Chat.ReadWrite` scope Blick already holds, but creating the chat is `POST /chats`, which needs `Chat.Create` — a scope Blick does not hold today, so this one costs a new consent prompt. For a one-on-one chat the member set is idempotent, so the work is resolving and reusing the existing thread with that person rather than spawning a duplicate.

The other cost is binding the picked contact to an Azure AD identity, since `POST /chats` names members by `userPrincipalName` or id. When the contact's email equals their UPN and they are in the tenant, Blick binds directly with no extra lookup; aliases, or a personal contact who is not a Teams user at all, either need a directory resolve or cannot be chatted. The natural pairing is compose-email below, which needs no new scope, so a reasonable phasing is email first, chat once the `Chat.Create` consent is worth adding.

## Compose and send a new email

The mail sibling of initiate-a-chat, and the cheaper of the two: a compose surface that sends a fresh message rather than only replying into a thread. Recipients come from the system Contacts picker (no `People.Read`, no permission prompt) or a typed address, with a subject, an optional cc and bcc, and file attachments. It sends through `Mail.Send`, which Blick already holds, so it costs no new scope. The real work is the compose UI itself; the Graph side is a single send call. The one wrinkle is attachments: an inline attachment is fine up to a few megabytes, but past Graph's inline ceiling a large file needs a chunked upload session, so attaching a file is a bit more than one request. Being scope-free, it is the sensible first compose feature to ship.

## Forward an email

Forward an unread message to new recipients with an optional note. Graph's `POST /messages/{id}/forward` builds the "Fwd:" subject and quotes the original body server-side, so Blick only collects recipients (Contacts picker) and a comment; it rides `Mail.ReadWrite` and `Mail.Send`, both held, so no new scope. This is a thin verb on top of a message already in hand, which makes it small and high-value. Leave the original unread by default, since forwarding something for input is not the same as being done triaging it — the user marks it read when they actually are.

## Recipient type-ahead in the composer

As the user types a fragment into To, Cc, or Bcc, offer tappable address matches beneath the field and complete the in-progress token on tap. This is the pattern Zirbe already runs in its composer: a keyed search with a generation counter so a slower earlier lookup cannot overwrite a newer keystroke's results, with matches shown as name-and-email rows. The build is a suggestion list under the focused field plus a debounced search; the composer already carries the field-fragment and append plumbing from the Contacts-picker work.

The cost is where the matches come from. Searching the device address book means `CNContactStore`, which, unlike the out-of-process `CNContactPickerViewController`, does need the Contacts permission and an `NSContactsUsageDescription` string, so type-ahead trades the picker's zero-permission property for a one-time prompt. A lighter alternative that needs no permission is to autocomplete only from sender identities already present in the fetched mail and chat streams: narrower, but free, and often enough for the people the user actually corresponds with. Reasonable phasing is recents-first with no permission, then optionally the full address-book search behind the prompt if recents prove too thin.

## Mail search

Find a message that is not in the unread front. A query by sender, subject, body text, or free words runs Graph's `$search` (KQL) over the mailbox, relevance-ranked and capped (the top 25 is plenty). It rides `Mail.ReadWrite`, already held, so no new scope; the Graph side is simple and the work is a query field and a results view. Blick is otherwise entirely unread-and-now, with no way to pull up a message from last week, so this is the biggest single gap in what the app can retrieve — worth ranking near the top of this group.

## Catch up on read and older messages

Blick shows only unread mail and chats in a live snapshot. A look-back view widens that to include already-read items and reaches up to thirty days back, so you can review what you have already seen or scroll the recent past, not just the unread edge. Same `Mail.ReadWrite` and `Chat.ReadWrite`, no new scope. The tradeoff is that it nudges Blick from a pure unread-triage surface toward a mail reader — more list and scroll UI, and a date-window control — so it earns its place only if the unread-only view proves too narrow once people are composing and searching from the app. Lower priority than search: search answers "find one specific thing", this answers "let me browse back a bit."

## Tap a summary section to open its full view

Make the glance a jumping-off point: tapping "Later today" opens the full-day calendar agenda, tapping the email section opens a full email list, and tapping the chats section opens a full chat list. Each tap trades the capped snapshot for the complete surface behind it, so this is the navigation spine that ties the retrieval features together — it is how the full-day agenda, the read-and-older look-back, and mail search get reached from the home screen instead of living as orphan views. No new scope; it is routing over data and views the other items already define. The tradeoff is the same altitude question they raise: every full view added moves Blick a step from glance toward full client, so this earns its place exactly as fast as those destination views do.

## Consolidated inbox: chats and emails in one list

Explore a single unified stream that interleaves unread email and Teams chats in one time-sorted list, offered as a setting rather than a replacement, so the user chooses between the current separate sections and one blended feed. It is closer to the single-conversation-feed feel Zirbe gives mail alone, but here it mixes two Graph sources. Both items already carry a received timestamp, so the merge and sort are straightforward; the work is a common row model that can render either kind and dispatch the right per-type actions, since an email reply (`/replyAll`, mark-read via message flags) and a chat reply (post into the thread, mark-read via the per-user read pointer) are different underneath a shared-looking row. No new scope, since it re-presents streams already fetched. Making it a settings toggle sidesteps the main risk, which is that one blended list may read noisier than two labeled sections once a subject-bearing email sits next to a bare chat line: ship both layouts and let the user pick, defaulting to the current split until the merged view proves itself.

## Full-day calendar agenda

Blick shows the next meeting plus "later today"; a full-day agenda would render the whole day — past events dimmed, the current one highlighted, a total-hours line. Extend the calendar data Blick already fetches (`Calendars.ReadWrite`, held) into that view, no new scope, squarely in posture. The only real question is altitude: whether a full agenda belongs in a status app when the native Calendar app exists, or whether "next plus later today" is the right amount. The lowest-stakes item in this group, worth it mainly if the day-at-a-glance view proves it wants more than the next event.

## Save and open email attachments

Blick renders attachments inline; this adds listing a message's attachments and saving one or opening it elsewhere. iOS has no user-facing filesystem, so the equivalent is routing an attachment to the share sheet or Files to save, and Quick Look or the owning app to open — over `Mail.ReadWrite`, already held, no new scope. Because inline rendering already covers the common "just let me see it" case, this is the most optional item here: it matters only for attachments the user wants to keep or hand to another app.

## Email and chat content on the Apple Watch

The watch already carries content, not just counts: the pushed snapshot includes the next meeting's subject and the "Later today" meeting subjects, while email and chat collapse to a number. That asymmetry is a display choice, not a posture limit, so close it by pushing message content the same way meetings already travel. Extend the WatchConnectivity snapshot with the top unread emails (sender, subject, and optionally Graph's `bodyPreview`) and the top pending chats (sender plus a short preview), and surface them in the glance beneath the meeting block, augmenting or replacing the count chips. This stays Tier 1 in every sense that matters: the watch still holds no token and makes no Graph call, it only receives more of the same non-credential data the phone already fetched, over the transport that already carries the meeting subjects. The payload is negligible, a few kilobytes of text against the application-context ceiling.

The one real tradeoff is display privacy rather than data governance. A count is safe to show at a glance; a sender and subject on the wrist are readable by anyone looking over your shoulder, so the open question is whether content is on by default or behind a setting, and how deep it goes — sender only, sender plus subject, or sender plus subject plus preview. The build is the snapshot shape plus a watch view; the decision is how much to reveal by default. Chat first is reasonable since a chat preview is already short, with email following the same pattern.

The natural extension, and the part that reads as "read my mail and chats on the watch" rather than just glance at them, is tapping a pushed item to open and read its full body on the wrist. This still fits Tier 1 without a token on the watch: the snapshot carries only the capped previews, and when the user opens one, the watch asks the phone for that message's full content over WatchConnectivity, the phone fetches it from Graph (the path it already uses for the preview sheet), and pushes the body back. The watch holds no token and makes no Graph call; it requests non-credential content on demand and the phone does the fetch, the same relay shape actions already use. The cost over the glance version is the on-demand round trip and a scrollable reader view on the watch, plus the reachability caveat that the phone must be in range to answer the request (out of range is exactly the Tier 2 case). Reading is naturally paired with the watch-side reply relay the action path already supports, so read-then-reply on the wrist becomes possible without the watch ever authenticating.

## Meeting-join nudge

A local notification a few minutes before the next meeting, with a one-tap join straight into Teams. Pure don't-miss-it utility and a daily driver, and fully in posture: the notification is scheduled on-device from the calendar data Blick already fetches, with no server and no push. The work is scheduling notifications off each upcoming event on the refresh cycle (the widget already builds a timeline of upcoming meeting starts, so the data shape exists) and wiring the notification action to the same Teams join path the meeting rows use.

The tradeoff is freshness: the scheduled set is only as current as the last background refresh, so a meeting accepted in the last hour might not have a nudge queued until the next refresh runs. That is acceptable for a reminder rather than a guarantee. It shares the background-refresh and notification scaffolding with the daily brief and the new-message nudge below, so the three are best costed as one investment.

## Morning daily brief

A once-a-day notification that opens the workday: next meeting plus how many remain, unread email and chat counts, and current presence, the same headline `WorkdaySummaryIntent` already speaks, delivered unprompted in the morning. Catch-up framing, and it pairs with the voice work since tapping in could drop straight into the read-aloud flow. In posture as a scheduled local notification fired off a background refresh, no backend.

The open decisions are timing and trigger: a fixed user-set time, or keyed to the first calendar event of the day or the start of Outlook working hours (the same `workingHours` object the auto-presence idea reads). Like the meeting nudge it depends on a fresh-enough background refresh, and it shares that plumbing with the nudge above and the new-message poll below.

## Best-effort new-message nudge

The in-posture answer to "tell me when something lands." A `BGAppRefreshTask` polls Graph on the cadence iOS grants and raises a local notification when new unread mail or a new chat has appeared since the last check. It reuses the background-refresh and notification scaffolding of the two ideas above, so once that exists this is largely a diff against the prior snapshot plus a notification.

The honest caveat, and the reason to write it down rather than expect more, is that timing is entirely OS-controlled and best-effort, not real-time. iOS decides when, and whether, a background refresh runs based on usage and power, so this is "you find out before too long," not "you find out the instant it arrives." That ceiling is structural: the real-time alternative, APNs or Graph change-notification webhooks, requires a server endpoint, which is a backend and a hard stop in the privacy posture. Poll plus local notification is as close as Blick gets without crossing that line, so this entry exists partly to keep real-time push from being re-proposed.

## Automatic presence: off-hours and iOS Focus

Set presence to Offline automatically when the user is off, releasing it when they are back, so they stop showing as around after hours without touching the manual toggle. Two complementary signals can drive this, and they compose as a precedence stack rather than two writers fighting over presence: a manual pick wins until it expires or is cleared (top), then an active linked Focus, then the Outlook-hours baseline, then normal auto.

This is additive, not a duplication of native behavior. M365 derives presence from Teams activity, the calendar (a busy event shows "In a meeting"), and automatic replies (Out of Office), but nothing native sets presence Offline by working hours. Viva Insights quiet time mutes notifications outside hours and does not change presence, so Blick's distinct niche is narrow and clear: controlling how the user appears to colleagues after hours.

Signal one, the passive baseline, is Outlook working hours. Read the `workingHours` object on `/me/mailboxSettings` (working days, start and end times, time zone) and set Offline outside that window, releasing it inside, on the background refresh cycle. This needs no new scope: Blick already holds `MailboxSettings.ReadWrite` for the Out-of-Office toggle, and working hours live on the same resource. Three constraints come with the data. It is a single window applied to every working day, so per-day hours (Monday to Thursday nine to five, Friday nine to noon) cannot be expressed. The time zone arrives as a Windows zone name ("Pacific Standard Time"), not an IANA identifier, so it needs the Windows-to-IANA mapping or the times requested in a known zone via the `Prefer: outlook.timezone` header. And there is no unset state: an untouched mailbox still returns a default (typically Monday to Friday, eight to five), so "deliberately set" and "never configured" look identical.

Signal two, the deliberate override, is an iOS Focus filter (`SetFocusFilterIntent`, iOS 16+). Blick defines a filter the user attaches to whatever Focus means off (or on) to them, in Settings, choosing the presence it applies. When that Focus activates the system runs the filter and Blick sets the configured presence; when it deactivates, Blick releases. Because the user configures each Focus, this overrides the hours baseline in both directions: a personal Focus at lunch sets Offline during a work hour, a work Focus in the evening pulls back online during off-hours. Focus state syncs across iPhone, Watch, and Mac, so toggling a Focus on the watch propagates to the phone, which runs the filter and makes the Graph call. That delivers watch control of presence on Tier 1, with no standalone-auth machinery, and it sidesteps the time-zone mapping entirely. The catch is that it only helps people who use Focus and have it set up, though that is a clean opt-in shape rather than a flaw: nothing happens until the filter is attached, and non-Focus users are unaffected.

Setting Offline uses `setUserPreferredPresence` (availability Offline, activity OffWork) and is still gated by an active presence session, the same PT1H session the manual toggle relies on. The privacy posture is unchanged, since it reads and writes only Microsoft Graph on the authenticated device. Both signals share the same background unknown to prove before building: the automation has to acquire a token silently and set presence with no UI open, the same path the widget and Control Center intents use, so it is feasible but should be verified on-device rather than assumed. Phasing: ship the Focus filter first (watch-controllable, no time-zone work, inherently opt-in, and the author uses Focus), prove the background presence-set on that simpler surface, then add Outlook hours as the passive baseline once the override behavior is trusted.

## Apple Watch Tier 2: independent auth on the wrist

The shipped watch app is Tier 1: it holds no token and makes no Graph call, fed non-credential data the phone pushes over WatchConnectivity. Tier 2 is an opt-in setting (default off) that lets the watch authenticate itself, hold its own device-bound token, and fetch fresh data and perform actions while standalone on cellular out of range of the phone. Even with it on, the phone stays the primary data path when in range; the watch self-fetches only when it cannot reach the phone, and for actions. The privacy line that must hold is that the watch's token is its own, acquired by its own sign-in and stored in the watch's own keychain with iCloud Keychain sync off; the phone's token is never copied to the watch and the watch never gets the phone's keychain access group. Implementation links CheckInGraph into the watch and gives it its own MSAL client and its own Entra redirect URI under the existing app registration.

The gate, and the reason this stays an idea rather than a commitment, is a spike. Microsoft sign-in on watchOS via `ASWebAuthenticationSession` (a full web login with MFA on the watch screen) is feasible on paper but unproven on this hardware, and even if it works, enterprise Conditional Access can reject the watch as a non-compliant device per tenant. So the tradeoff is real engineering and auth-surface risk for a capability most users will not need, and the feature must fall back silently to Tier 1 whenever the spike, the runtime sign-in, or a tenant blocks it. Prove the spike on-device before building anything else. David likes the app as it is and sees no current need, so this is parked, not scheduled.
