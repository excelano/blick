# CheckIn: Potential Features

Ideas under consideration, not yet committed to the build. Each entry notes the rough approach and the main tradeoff so we can judge priority later. Shipped functionality lives in FEATURES.md.

## Automatic presence: off-hours and iOS Focus

Set presence to Offline automatically when the user is off, releasing it when they are back, so they stop showing as around after hours without touching the manual toggle. Two complementary signals can drive this, and they compose as a precedence stack rather than two writers fighting over presence: a manual pick wins until it expires or is cleared (top), then an active linked Focus, then the Outlook-hours baseline, then normal auto.

This is additive, not a duplication of native behavior. M365 derives presence from Teams activity, the calendar (a busy event shows "In a meeting"), and automatic replies (Out of Office), but nothing native sets presence Offline by working hours. Viva Insights quiet time mutes notifications outside hours and does not change presence, so CheckIn's distinct niche is narrow and clear: controlling how the user appears to colleagues after hours.

Signal one, the passive baseline, is Outlook working hours. Read the `workingHours` object on `/me/mailboxSettings` (working days, start and end times, time zone) and set Offline outside that window, releasing it inside, on the background refresh cycle. This needs no new scope: CheckIn already holds `MailboxSettings.ReadWrite` for the Out-of-Office toggle, and working hours live on the same resource. Three constraints come with the data. It is a single window applied to every working day, so per-day hours (Monday to Thursday nine to five, Friday nine to noon) cannot be expressed. The time zone arrives as a Windows zone name ("Pacific Standard Time"), not an IANA identifier, so it needs the Windows-to-IANA mapping or the times requested in a known zone via the `Prefer: outlook.timezone` header. And there is no unset state: an untouched mailbox still returns a default (typically Monday to Friday, eight to five), so "deliberately set" and "never configured" look identical.

Signal two, the deliberate override, is an iOS Focus filter (`SetFocusFilterIntent`, iOS 16+). CheckIn defines a filter the user attaches to whatever Focus means off (or on) to them, in Settings, choosing the presence it applies. When that Focus activates the system runs the filter and CheckIn sets the configured presence; when it deactivates, CheckIn releases. Because the user configures each Focus, this overrides the hours baseline in both directions: a personal Focus at lunch sets Offline during a work hour, a work Focus in the evening pulls back online during off-hours. Focus state syncs across iPhone, Watch, and Mac, so toggling a Focus on the watch propagates to the phone, which runs the filter and makes the Graph call. That delivers watch control of presence on Tier 1, with no standalone-auth machinery, and it sidesteps the time-zone mapping entirely. The catch is that it only helps people who use Focus and have it set up, though that is a clean opt-in shape rather than a flaw: nothing happens until the filter is attached, and non-Focus users are unaffected.

Setting Offline uses `setUserPreferredPresence` (availability Offline, activity OffWork) and is still gated by an active presence session, the same PT1H session the manual toggle relies on. The privacy posture is unchanged, since it reads and writes only Microsoft Graph on the authenticated device. Both signals share the same background unknown to prove before building: the automation has to acquire a token silently and set presence with no UI open, the same path the widget and Control Center intents use, so it is feasible but should be verified on-device rather than assumed. Phasing: ship the Focus filter first (watch-controllable, no time-zone work, inherently opt-in, and the author uses Focus), prove the background presence-set on that simpler surface, then add Outlook hours as the passive baseline once the override behavior is trusted.

## Apple Watch Tier 2: independent auth on the wrist

The shipped watch app is Tier 1: read-only, fed non-credential status data the phone pushes over WatchConnectivity, holding no token and making no Graph call. Tier 2 is an opt-in setting (default off) that lets the watch authenticate itself, hold its own device-bound token, and fetch fresh data and perform actions while standalone on cellular out of range of the phone. Even with it on, the phone stays the primary data path when in range; the watch self-fetches only when it cannot reach the phone, and for actions. The privacy line that must hold is that the watch's token is its own, acquired by its own sign-in and stored in the watch's own keychain with iCloud Keychain sync off; the phone's token is never copied to the watch and the watch never gets the phone's keychain access group. Implementation links CheckInGraph into the watch and gives it its own MSAL client and its own Entra redirect URI under the existing app registration.

The gate, and the reason this stays an idea rather than a commitment, is a spike. Microsoft sign-in on watchOS via `ASWebAuthenticationSession` (a full web login with MFA on the watch screen) is feasible on paper but unproven on this hardware, and even if it works, enterprise Conditional Access can reject the watch as a non-compliant device per tenant. So the tradeoff is real engineering and auth-surface risk for a capability most users will not need, and the feature must fall back silently to Tier 1 whenever the spike, the runtime sign-in, or a tenant blocks it. Prove the spike on-device before building anything else. David likes the app as it is and sees no current need, so this is parked, not scheduled.
