# Blick: Potential Features

Ideas under consideration, not yet committed to the build. Each entry notes the rough approach and the main tradeoff so we can judge priority later. Shipped functionality lives in FEATURES.md.

## Automatic presence: off-hours and iOS Focus

Set presence to Offline automatically when the user is off, releasing it when they are back, so they stop showing as around after hours without touching the manual toggle. Two complementary signals can drive this, and they compose as a precedence stack rather than two writers fighting over presence: a manual pick wins until it expires or is cleared (top), then an active linked Focus, then the Outlook-hours baseline, then normal auto.

This is additive, not a duplication of native behavior. M365 derives presence from Teams activity, the calendar (a busy event shows "In a meeting"), and automatic replies (Out of Office), but nothing native sets presence Offline by working hours. Viva Insights quiet time mutes notifications outside hours and does not change presence, so Blick's distinct niche is narrow and clear: controlling how the user appears to colleagues after hours.

Signal one, the passive baseline, is Outlook working hours. Read the `workingHours` object on `/me/mailboxSettings` (working days, start and end times, time zone) and set Offline outside that window, releasing it inside, on the background refresh cycle. This needs no new scope: Blick already holds `MailboxSettings.ReadWrite` for the Out-of-Office toggle, and working hours live on the same resource. Three constraints come with the data. It is a single window applied to every working day, so per-day hours (Monday to Thursday nine to five, Friday nine to noon) cannot be expressed. The time zone arrives as a Windows zone name ("Pacific Standard Time"), not an IANA identifier, so it needs the Windows-to-IANA mapping or the times requested in a known zone via the `Prefer: outlook.timezone` header. And there is no unset state: an untouched mailbox still returns a default (typically Monday to Friday, eight to five), so "deliberately set" and "never configured" look identical.

Signal two, the deliberate override, is an iOS Focus filter (`SetFocusFilterIntent`, iOS 16+). Blick defines a filter the user attaches to whatever Focus means off (or on) to them, in Settings, choosing the presence it applies. When that Focus activates the system runs the filter and Blick sets the configured presence; when it deactivates, Blick releases. Because the user configures each Focus, this overrides the hours baseline in both directions: a personal Focus at lunch sets Offline during a work hour, a work Focus in the evening pulls back online during off-hours. Focus state syncs across iPhone, Watch, and Mac, so toggling a Focus on the watch propagates to the phone, which runs the filter and makes the Graph call. That delivers watch control of presence on Tier 1, with no standalone-auth machinery, and it sidesteps the time-zone mapping entirely. The catch is that it only helps people who use Focus and have it set up, though that is a clean opt-in shape rather than a flaw: nothing happens until the filter is attached, and non-Focus users are unaffected.

Setting Offline uses `setUserPreferredPresence` (availability Offline, activity OffWork) and is still gated by an active presence session, the same PT1H session the manual toggle relies on. The privacy posture is unchanged, since it reads and writes only Microsoft Graph on the authenticated device. Both signals share the same background unknown to prove before building: the automation has to acquire a token silently and set presence with no UI open, the same path the widget and Control Center intents use, so it is feasible but should be verified on-device rather than assumed. Phasing: ship the Focus filter first (watch-controllable, no time-zone work, inherently opt-in, and the author uses Focus), prove the background presence-set on that simpler surface, then add Outlook hours as the passive baseline once the override behavior is trusted.

## Curated people list and message filtering

A short, user-curated list of the people who matter most, maintained on the device, closer to blick-cli's small list than Zirbe's full Contacts access. Its first payoff is catch-up: filter the mail and Teams streams Blick already fetches down to just those people, so "what did my key people send today" is one tap. This needs no new Graph scope, because the filtering happens locally over messages already in hand, matched by sender identity. The list itself is the only new state, stored on-device (UserDefaults, or the app group so the widget could read it later), holding just enough to identify a person such as the Graph user id plus a display name and email or UPN, with no backend and nothing synced off the device, so the privacy posture is unchanged.

The open design question is how a person gets onto the list. The zero-new-scope path is to seed and add from people who already appear in the fetched mail and chats, since those streams already carry sender identities; the user stars someone they already correspond with, which fits the catch-up framing because your important people are usually people you already hear from. Adding someone you have never messaged would need a directory lookup, either `/me/people` (`People.Read`, relevance-ranked, the close match to a short list) or a broader `/users` search scope, and that is the one place this would cost a new consent prompt. So phase it: ship the list seeded from existing correspondents plus local filtering first with no new scope, and treat directory search as a later add-on only if curating from the streams proves too narrow. This primitive is also the foundation the initiate-a-chat feature below builds on.

## Initiate a Teams chat (new message)

A compose surface that lets the user start a new Teams chat from Blick rather than only replying into threads that already carry pending activity. It builds directly on the curated people list above: pick a person from the list, type a message, send. Creating or reusing a one-on-one chat is `POST /chats` and posting the first message is `POST /chats/{id}/messages`, and both are already covered by the `Chat.ReadWrite` scope Blick holds today for the reply surface, so this needs no new permission. For a one-on-one chat the member set is effectively idempotent, so the work is resolving and reusing the existing thread with that person rather than spawning a duplicate; group chats would be a later extension. Email initiation is the obvious sibling, since `Mail.Send` is already held, but the deliberate call is chat first, email later.

The real cost here is not Graph, it is product direction. This is an initiate/compose feature, exactly the category we decided on 2026-06-23 to keep in blick-cli and out of Blick iOS, on the reasoning that iOS is for catching up while the keyboard-driven CLI is where firing off a message belongs ([[blick-tools-diverge]]). The counter-argument, and the reason this is worth capturing now, is that the people-list primitive earns its place on filtering alone, so the marginal step to initiate is small once the list exists, and a person whose message you have just read is a person you may want to start a fresh thread with. Shipping only the filter half and holding the line on divergence is a perfectly coherent outcome; this entry exists so the option is on the table, not because it is decided.

## Email and chat content on the Apple Watch

The watch already carries content, not just counts: the pushed snapshot includes the next meeting's subject and the "Later today" meeting subjects, while email and chat collapse to a number. That asymmetry is a leftover of the glance framing, not a posture limit, so close it by pushing message content the same way meetings already travel. Extend the WatchConnectivity snapshot with the top unread emails (sender, subject, and optionally Graph's `bodyPreview`) and the top pending chats (sender plus a short preview), and surface them in the glance beneath the meeting block, augmenting or replacing the count chips. This stays Tier 1 in every sense that matters: the watch still holds no token and makes no Graph call, it only receives more of the same non-credential data the phone already fetched, over the transport that already carries the meeting subjects. The payload is negligible, a few kilobytes of text against the application-context ceiling.

The one real tradeoff is display privacy rather than data governance. A count is safe to show at a glance; a sender and subject on the wrist are readable by anyone looking over your shoulder, so the open question is whether content is on by default or behind a setting, and how deep it goes — sender only, sender plus subject, or sender plus subject plus preview. The build is the snapshot shape plus a watch view; the decision is how much to reveal by default. Chat first is reasonable since a chat preview is already short, with email following the same pattern.

## Voice catch-up: read unread aloud and act by voice

The killer feature taken to its end: a Siri flow that reads the unread queue aloud, sender and subject and a short preview per item, and after each one offers to reply, mark it read, or move to the next, all spoken. This is catch-up for the moments the app and the CLI both fail you, driving or walking or between meetings, and nothing in Zirbe or blick-cli does it. It stays in posture because the phone already holds the mail and chats; the voice layer reads what is already fetched and routes a reply or a mark-read through the same paths the app uses.

The real constraint is the App Intents spoken-phrase ceiling, which Blick already sits near. So the design rides the existing `WorkdaySummaryIntent` progressive-disclosure shape rather than minting new top-level phrases: "What's my Blick" already drills from headline to who-and-how-many to each remaining meeting, and reading the items aloud is the natural next level of that same disclosure, with reply and mark-read as the spoken follow-ups. Dictated replies go through Siri's own dictation. Phase it: read-aloud first, since it is the hard part and the main value, then spoken reply. Relationship to the Earshot concept ([[teams-voice-app-concept]]): that is ambient, per-contact Teams voices as messages arrive, while this is on-demand "read me what I missed" triage, a different enough experience to belong in Blick.

## Meeting-join nudge

A local notification a few minutes before the next meeting, with a one-tap join straight into Teams. Pure don't-miss-it utility and a daily driver, and fully in posture: the notification is scheduled on-device from the calendar data Blick already fetches, with no server and no push. The work is scheduling notifications off each upcoming event on the refresh cycle (the widget already builds a timeline of upcoming meeting starts, so the data shape exists) and wiring the notification action to the same Teams join path the meeting rows use.

The tradeoff is freshness: the scheduled set is only as current as the last background refresh, so a meeting accepted in the last hour might not have a nudge queued until the next refresh runs. That is acceptable for a reminder rather than a guarantee. It shares the background-refresh and notification scaffolding with the daily brief and the new-message nudge below, so the three are best costed as one investment.

## Morning daily brief

A once-a-day notification that opens the workday: next meeting plus how many remain, unread email and chat counts, and current presence, the same headline `WorkdaySummaryIntent` already speaks, delivered unprompted in the morning. Catch-up framing, and it pairs with the voice work since tapping in could drop straight into the read-aloud flow. In posture as a scheduled local notification fired off a background refresh, no backend.

The open decisions are timing and trigger: a fixed user-set time, or keyed to the first calendar event of the day or the start of Outlook working hours (the same `workingHours` object the auto-presence idea reads). Like the meeting nudge it depends on a fresh-enough background refresh, and it shares that plumbing with the nudge above and the new-message poll below.

## Best-effort new-message nudge

The in-posture answer to "tell me when something lands." A `BGAppRefreshTask` polls Graph on the cadence iOS grants and raises a local notification when new unread mail or a new chat has appeared since the last check. It reuses the background-refresh and notification scaffolding of the two ideas above, so once that exists this is largely a diff against the prior snapshot plus a notification.

The honest caveat, and the reason to write it down rather than expect more, is that timing is entirely OS-controlled and best-effort, not real-time. iOS decides when, and whether, a background refresh runs based on usage and power, so this is "you find out before too long," not "you find out the instant it arrives." That ceiling is structural: the real-time alternative, APNs or Graph change-notification webhooks, requires a server endpoint, which is a backend and a hard stop in the privacy posture. Poll plus local notification is as close as Blick gets without crossing that line, so this entry exists partly to keep real-time push from being re-proposed.

## Apple Watch Tier 2: independent auth on the wrist

The shipped watch app is Tier 1: it holds no token and makes no Graph call, fed non-credential data the phone pushes over WatchConnectivity. Tier 2 is an opt-in setting (default off) that lets the watch authenticate itself, hold its own device-bound token, and fetch fresh data and perform actions while standalone on cellular out of range of the phone. Even with it on, the phone stays the primary data path when in range; the watch self-fetches only when it cannot reach the phone, and for actions. The privacy line that must hold is that the watch's token is its own, acquired by its own sign-in and stored in the watch's own keychain with iCloud Keychain sync off; the phone's token is never copied to the watch and the watch never gets the phone's keychain access group. Implementation links CheckInGraph into the watch and gives it its own MSAL client and its own Entra redirect URI under the existing app registration.

The gate, and the reason this stays an idea rather than a commitment, is a spike. Microsoft sign-in on watchOS via `ASWebAuthenticationSession` (a full web login with MFA on the watch screen) is feasible on paper but unproven on this hardware, and even if it works, enterprise Conditional Access can reject the watch as a non-compliant device per tenant. So the tradeoff is real engineering and auth-surface risk for a capability most users will not need, and the feature must fall back silently to Tier 1 whenever the spike, the runtime sign-in, or a tenant blocks it. Prove the spike on-device before building anything else. David likes the app as it is and sees no current need, so this is parked, not scheduled.
